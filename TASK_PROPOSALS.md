# 改善タスク提案（コードベース概要レビュー結果）

対象リポジトリを俯瞰し、既存のテスト・既知課題・実装構造から、優先度と着手しやすさのバランスが良い 4 タスクを提案します。

## 1) 入力ミス修正タスク（typo / 編集ミス）

### タスク
`KNOWN_ISSUES.md` と `TEST_REPORT.md` に残っている **マージコンフリクト記号**（`<<<<<<<`, `=======`, `>>>>>>>`）を解消する。

### 背景
両ファイルに競合マーカーが残存しており、ドキュメントの可読性と信頼性を大きく下げています。CI で markdown lint を導入した場合にも将来的な失敗要因になります。

### 完了条件
- 2 ファイルから競合マーカーが完全になくなる。
- 内容を 1 つの正しい版に統合し、日付と記述の整合性を取る。

---

## 2) バグ修正タスク

### タスク
バンドル対象エントリーポイントの不整合を修正する（`package.json` の `build` と `start` の参照先統一）。

### 背景
- `build` は `server/_core/index.ts` を `dist/index.js` に出力。
- `start` も `dist/index.js` を実行。
- 一方で、リポジトリには `server/index.ts` と `server/_core/index.ts` が共存しており、実運用でどちらが正なのか曖昧になりやすい構成です。

この曖昧さは、修正が片側にだけ入って本番に反映されない「実質バグ」を生みやすいです。

### 完了条件
- 正式なサーバー起点を 1 つに決定し、`build/dev/start` の参照を統一。
- README か運用ドキュメントにエントリーポイント方針を明記。
- `pnpm build && pnpm start` で起動確認。

---

## 3) コメント/ドキュメント矛盾修正タスク

### タスク
`KNOWN_ISSUES.md` と `TEST_REPORT.md` の内容矛盾（「リモート未設定」「build warning の種類」「テスト補足内容」）を整合化する。

### 背景
同日付の報告書に複数版が混在しており、現状判断に使うには不正確です。特に運用引き継ぎ時に誤解の原因になります。

### 完了条件
- 重複・矛盾を統合した単一の記述へ更新。
- 各レポートの「生成日」「実行コマンド」「結果」「既知制約」を明確に分離。
- 必要なら `IMPLEMENTATION_REPORT.md` から相互参照リンクを追加。

---

## 4) テスト改善タスク

### タスク
ドキュメント健全性チェックをテスト/CI に追加する（最低限、競合マーカー検出テスト）。

### 背景
今回のような競合マーカー残存は、既存の `pnpm test`（Vitest）だけでは検知できません。軽量な静的チェックを追加すると再発防止に効果的です。

### 完了条件
- `scripts/check-conflicts.sh` などを追加し、`<<<<<<<|=======|>>>>>>>` を検出したら失敗。
- `package.json` に `test:docs`（または `check:repo`）を追加。
- CI 実行対象へ組み込み（ローカルでも `pnpm run test:docs` が失敗/成功を確認可能）。

---

## 優先実施順
1. 入力ミス修正（即時）
2. ドキュメント矛盾修正（即時）
3. テスト改善（再発防止）
4. エントリーポイント統一バグ修正（影響調査込みで実施）
