# 月締め〜請求〜入金の一連フロー（正本）

> オーナー(biguoki@gmail.com)確認済み。今後この流れを繰り返し説明せずに済むよう恒久記録する。
> 現状の財務まわりは7画面に分散（請求書/月締めV2/作業員請求書V2/締め管理/確認表PDF/支払管理/入金管理）していて「整理されていない・機能を足しただけ」の状態。この流れに沿って再構築する。

## 大原則（重要な訂正）
- **「月締めを進む」を押した = 出面を確認し出勤日数が確定した、ということ**。→ 改めての「出面確認」ステップは不要（会社の確認ベースは記入済みの出面そのもの）。
- **交通費は月締めで最重要情報**。交通費機能は維持。作業員が交通費を入れないと請求書を発行できない。
- **会社が「作業員の月締め」画面で請求書を確認する機能は不要**（作業員が交通費を入れて初めて請求書が出るため）。
- **柔軟性が最重要**（管理者が代行できる・例外に対応できる）。

## 作業員側フロー（全作業員 共通）
1. 出面の確認と確定（＝「進む」を押すこと自体が確定）
2. 交通費の有無・記載・領収書アップロード
3. 1と2の整合確認 ＋ 請求書プレビュー ＋ 金額確認（自動計算主体なのでほぼ誤りなし）
   - 3-2. 是正があれば是正して内容を確定
4. 領収書 ＋ 日報（請求の元）＋ 請求書 を会社へ提出（＝請求を確定）

## 会社側フロー(1)：プロジェクト月締め
1. プロジェクト確認
2. 関わる作業員の提出状況確認。未提出は催促
3. 確定情報・アップロード書類を見て是正の有無を確認。是正あれば理由を指摘して差し戻し。
   （未提出・軽微な是正は管理者が代行した方が早い場合あり＝柔軟対応が必要）。問題なければOK
4. 全作業員をOKにする（＝プロジェクトの月締め完了）
- 全プロジェクトを完了にする。完了したプロジェクトは取引先別に自動で割り当て・選別される。

## 会社側フロー(2)：取引先請求
- ある取引先の**全プロジェクトが月締め完了 → 取引先請求書を自動生成（＝請求の確定）**。ほぼ誤りなし。例外は経費追加くらい。
- プレビューして請求を確定。

## 保存・管理
- 作業員から提出された請求書を保存。月別に閲覧可。
- **支払い待ち / 支払い済み** を管理側・作業員側の両方で閲覧可。
- 取引先へ提出した請求書と入金（支払い）確認も管理。
- 会計ソフトを参考にした請求書管理を検討。

## 確定した設計判断（オーナー回答済み）
- **承認の粒度**：作業員ごとに承認/差し戻し → 全員承認でプロジェクト自動完了（手動完了も可）。
- **管理者の代行**：未提出・軽微な是正のとき、管理者が作業員に代わって交通費入力〜提出まで**完全代行**できる。
- **差し戻し通知**：アプリ内のみ（作業員ホームに差し戻し表示＋バッジ）。メール/LINEは無し。
- **前借り（案A）**：作業員ごとに「前借り/立替 台帳（残高）」を持ち、支払い時に**自動で相殺提案＋履歴**。
- **入金/請求管理**：**部分入金（分割払い）は基本扱わない**（全額単位）。**会計ソフト（freee/MF）向けCSV出力は用意する**。
- **画面統合（IA）**：管理者3画面（①月締め ②請求[売上] ③支払[作業員]）＋作業員1画面（マイ月締め）＋ダッシュボード。不要画面（確認表PDF単独・作業員請求書V2単独 等）は統合して削除。
