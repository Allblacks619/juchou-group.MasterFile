# 権限×情報の整理・分析（正本）

> 2026-07-20 権限監査（オーナー指示による全面棚卸し）。今後の権限まわりの変更はこの文書を正とする。
> きっかけ: 作業員のスマホに入金管理（取引先請求額）が表示されていた／従業員管理で誰がどの権限か分からなかった。

## 1. ロールの定義（5段階）

| ロール | 表示名 | できること（サーバー実効） |
|---|---|---|
| super_admin | 統括管理者 | 全て＋ロール変更・パスワード再発行・従業員一括削除 |
| admin | 管理者 | 財務・従業員・現場・単価の全管理（ロール変更は不可） |
| manager | 責任者 | admin とほぼ同じ管理画面（会社設定の更新・削除系など一部は admin 以上） |
| worker | 作業員 | 自分の情報のみ: マイプロフィール／月締め提出／作業日報／同じ現場の出面 |
| guest | ゲスト | 最小限（現場ビジョンの閲覧系など） |

- **注意: manager は財務が全部見える**。取引先請求額・全作業員の支払額・前借り・単価すべて閲覧可能（`leaderOrAdminProcedure` = super_admin/admin/manager）。「現場リーダーに付ける軽い権限」ではない。
- **旧 "leader" ロール**: DB enum に残っているがサーバーは worker 扱いに正規化する。招待作成時は manager に変換される。新規に使わないこと。
- ロールは **users テーブル（アカウント）** に付く。従業員（employees）とは `employees.userId` で紐づく。アカウント未連携の従業員にはロールが無い。

## 2. 2026-07-20 の事象の説明（スクショ3枚）

1. **作業員のスマホに入金管理・請求書画面が出た**: ロールを一括変更（manager）した結果、その作業員アカウントが財務メニューを取得したもの。作業員ロールのままならナビにも出ず、URL直打ちもクライアント（RoleGuard）とサーバー（403）の二重で遮断される。
2. **「締めが完了していないため請求作成できません（open）」**: 取引先請求書の自動作成ダイアログ（請求書管理画面）の正しいガード。現場単位の月締め（projectClosings）が open のうちは取引先請求書を作らせない仕様であり、バグではない。作業員本人の月締めは「月締め提出」（/app/my-closing）が正しい導線。
3. **従業員管理で権限が分からない**: 当時はロール表示が存在しなかった（表示は super_admin 専用の「パスワード再発行」画面のみ）。→ 今回、従業員管理に権限列＋個別変更UIを実装済み。

## 5. 運用ルール（オーナー向け）

1. **作業員に manager を付けない**。manager=財務全部見える権限。現場のリーダー役でも、財務を見せないなら worker のまま。
2. ロール変更は従業員管理画面で。**一括変更の前に選択バーの「現在ロール内訳」を確認**。個別変更は行の権限バッジから。
3. 「未連携」バッジの従業員はアカウントが無い（ロール変更不可）。招待からアカウントを作ると worker で始まる。
4. 作業員への案内は「**月締め提出**」（メニューの上部）。「請求書」「月締めV2」「締め管理」は会社側の画面。

## 6. 未対応・要判断の論点

1. **manager の権限範囲**: 現状 manager ≒ admin。財務を見せない中間ロール（現場管理はできるが金額は見えない）が必要なら新設が必要（現状はない）。
2. **employee.list の情報量**: manager 以上は全従業員の銀行口座・在留カード番号まで閲覧できる。人事情報を admin 以上に絞るならサーバー側で返却カラムの絞り込みが必要。
3. **旧 "leader" ロールの掃除**: DB に leader のアカウントが残っていれば実効は worker。従業員管理のバッジは「作業員」と表示される。DB enum からの削除は別途マイグレーション判断。
4. **紛らわしい画面名**: 「月締め提出（作業員）」「月締めV2（会社）」「締め管理（会社）」の名称整理は画面統合フェーズで実施予定。
