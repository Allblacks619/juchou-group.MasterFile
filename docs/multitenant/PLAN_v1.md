# マルチテナント化・会社間連携 構想プラン v1（正本ドラフト）

- 作成日: 2026-07-12
- ステータス: **オーナーレビュー待ち（構想段階・本体コード未変更・シニアアーキテクト役による審議レビュー15件反映済み）**
- 目的: 本システムを「1社専用」から「協力会社同士（二次⇔三次⇔四次）が請求書・出面・作業員名簿をスマートにやり取りできる建設業特化プラットフォーム」へ進化させるための設計正本。
- 大原則: **本番は現状のまま運用継続。マルチテナント機能はフィーチャーフラグ（`MULTI_TENANT`、既定 off）+ 別DBシード + 自動テストで裏側検証**し、十分に固まるまで本番挙動を一切変えない。

---

## 0. オーナーの考え（合意済みの前提）

2026-07-12 のヒアリングで確定した方針:

1. **相手（例: 元請）も正式テナント**にする。相手にも月締めの便利さを体験してもらう将来像。
2. ただし特殊パターンがある（オーナーの実取引そのもの）:
   - 自社（二次）が**元請の現場のプロジェクト管理を代行**している。
   - その現場には**元請の職人がゲストとして**入っている（genba のゲスト作業員）。
   - それでも**請求書は会社ごとに別**（自社→元請への請求書を発行する義務がある）。
   - 元請は自社からの請求を取り込んだうえで、**さらに上位の取引先へ請求書を出す**（多段請求チェーン）。
3. 第1弾マイルストーンは **(a) 請求書提出＋出面確認** と **(b) 作業員名簿・資格書提出** の2本。
4. 検証は**別DB＋ローカル/テスト検証**（本番非接触）。架空3社（実取引を模す）でシーディングしてシミュレーション。
5. 入場前の名簿+資格書提出（グリーンファイル的な流れ）は実務上の必須業務であり、連携対象。

## 1. 現状分析サマリ（3系統の並行調査結果）

### 1.1 テナント分離の現状 — 完全なシングルテナント

- `companyId`/`tenantId` に相当する列は**全51テーブルで0件**。`companyProfile` は `limit(1)` で常に1行を返すシングルトン（`server/db.ts:625-642`）。
- セッション JWT は `{ openId, appId, name }` のみ（`server/_core/sdk.ts:21-25`）。**会社識別子がセッションに存在しない**。ctx 生成は `server/_core/context.ts:11-28` の1箇所。
- 認可はロール（縦）のみで、会社（横）の分離チェックはゼロ。`adminProcedure` を通れば全データにアクセスできる。
- `db.getAll*()` 系（clients/projects/employees/invoices/users/invitations/rates/advances 等）が**無条件全件返却**。マルチテナント化ではこの全てに会社フィルタが要る。
- `clients`（取引先）は名前・住所だけの**静的マスタ行**。アカウント・ログイン・相互参照の概念なし。
- `invitations` は社内ロール専用。他社を招く導線は存在しない。
- 監査ログ `auditLogs` にも会社列なし。

**壊れやすい箇所トップ10**（詳細は付録A）: ①セッションに会社IDなし ②companyProfile単数固定 ③getAll*の全件返却 ④横分離ゼロの認可 ⑤clientsのマスタ行止まり ⑥社内専用の招待 ⑦openId/loginIdのグローバル一意制約 ⑧genba26テーブルのスコープ皆無 ⑨公開トークンのテナント貫通 ⑩監査/締め/請求のint直参照。

### 1.2 請求・月締めフロー — 「会社間」の器はないが、雛形は社内に完成している

- 月締めは V1（`project_closings` 系）と V2（`monthly_closing_v2_*` 系）が**ブリッジ併存**（`server/routers.ts:3712-3726`、`clientInvoiceV2Builder.ts:143-159`）。
- **作業員→会社の請求書は既に双方向ワークフローとして完成**: draft → submitted → returned/approved、差戻し理由必須、**提出時スナップショット凍結**（`workerInvoiceSnapshots`）、PDFはスナップショットからのみ生成、管理者代行、添付証憑パイプライン。**これをそのまま「会社→会社」に写像するのが最短経路**。
- 一方、**取引先請求書には受領側の状態機械が存在しない**。`invoices.status` の `sent` は手動セットのみで、実際の提出は PDF を系外（メール/手渡し）で送るだけ。**ここが会社間連携の最大の空白**。
- 入金は `invoices` 行への相乗り（`receivedAmount` 等）。「A社の入金 = B社の支払」という対称性を表現できない。
- 自社前提のハードコード: 社名「充寵グループ」が PDF 生成器・フロント等 20箇所以上（特に `pdfWorkerInvoice.ts:105-106` は**宛先を常に自社に固定**）。請求書番号採番はテナント非分離のグローバル連番。インボイス発行者判定は自社 companyProfile 固定。

### 1.3 genba・名簿・ゲスト — 会社間連携に転用できる資産が最も揃っている領域

- **genbaSiteWorkers**: 登録従業員/アカウント無し従業員/ゲスト（guestName）の3種を1つの安定IDに正規化済み。元請の職人は出面の guestName 経由で自動的に名簿へ現れる。
- **トークンベース共有が2系統完成**: `genbaWorkerLinks`（ログイン不要・role別編集権限・ソフト失効・期限・lastAccess）と `genbaShares`（scope付き閲覧専用・ホワイトリスト方式 `buildShareView`・秘匿NOT_FOUND）。セキュリティテストで漏洩禁止フィールドまで担保済み（`genba.security.test.ts`）。
- **名簿PDF資産**: `employees` + `qualifications`（証書ファイル付き）+ `documents`（在留/保険/CCUS等の documentType 網羅）+ `generateRosterPdf`（個票/一覧/束ね）で、建設業標準の作業員名簿が既にワンショット生成可能。
- 未整備: ゲストの一意性が `(siteId, guestName)` の**文字列一致依存**（同名衝突・表記ゆれに弱い）。ゲストに資格書を紐付ける経路がない。他社から名簿を「受け取る」側の機能がない。CCUS番号（`employees.careerUpNumber`）が名寄せに未活用。

## 2. 目標アーキテクチャ

### 2.1 基本コンセプト: 「テナント分離」+「コネクト層」の二層構造

```
┌─ テナントA（充寵 = 二次）─┐      ┌─ テナントB（元請）──────┐
│ users / employees        │      │ users / employees        │
│ projects / attendance    │      │ projects / attendance    │
│ invoices / closings ...  │      │ invoices / closings ...  │
└──────────┬───────────────┘      └───────────┬──────────────┘
           │        ★ コネクト層（会社間のみが通る橋）         │
           └──→ partner_links（取引関係の相互承認）←──────────┘
                partner_invoice_submissions（請求書提出箱）
                partner_roster_submissions（名簿・資格書提出箱）
                partner_attendance_views（出面確認の共有ビュー）
```

- **テナント内は完全分離**: 全業務テーブルに `companyId` を追加し、ctx で強制フィルタ。他社のデータは絶対に直接見えない。
- **会社間のやり取りは必ず「コネクト層」を経由**: 提出（submission）はスナップショット凍結された不変コピーとして相手の受領箱に入る。**相手のDBの生データを覗く方式は採らない**（単価・原価・従業員個人情報などの機密が漏れるため）。
- これは既存の勝ちパターンの再利用: 「作業員→会社」の提出フロー（スナップショット+状態機械）と、genba の「トークン共有」（ホワイトリスト払い出し）の合成。

### 2.2 テナント基盤（Phase 1 の中身）

1. **`companies` テーブル新設**（テナント master）。現行 `companyProfile` の内容を包含（社名/ロゴ/印影/振込先/インボイス登録番号/住所…）。既存の自社データは `companyId=1` として移行。
2. **`companyId` 列の追加**: users, employees, clients, projects, attendance, invoices+items, closings系（V1/V2全部）, payments, workerInvoices系, rates系, qualifications, documents, invitations, auditLogs, workerAdvances, passwordResetRequests, genba全テーブル。**既存行は全て `companyId=1` にバックフィル**するだけなので現行動作は不変。
3. **セッション/ctx**: `SessionPayload` に `companyId` を追加し、`createContext`（`context.ts:23`）で `ctx.companyId` を注入。既存トークンは companyId 未設定なら `1` にフォールバック（＝現行ユーザーのログインを壊さない）。
4. **クエリガード**: `db.getAll*()` を `companyId` 必須の関数に置換。移行期間は「`MULTI_TENANT=off` なら companyId=1 固定」で完全互換。**テナント混線を検出する自動テスト**（2社シードして相互不可視を総当たり検証）を先に書く。
5. **一意制約の方針（審議で修正）**: `loginId`/`openId` は**グローバル一意のまま維持**し、「1ユーザー = 1会社所属」を第1弾の前提とする。ログイン時点では companyId が未知のため（`getUserByLoginId` はグローバル検索、`routers.ts:1213`）、複合 unique 化するとログインが曖昧になる。複合 unique 化・複数社所属（membership テーブル）は、パスワードリセット・招待フローの再設計とセットの**独立した将来フェーズ**に隔離する。`guestName` 等の業務系 unique のみ `(companyId, x)` 複合化。なお `attendance_guest_proj_date`（`schema.ts:477`）のように NULL 許容列を含む unique は MariaDB では NULL 行が重複可という既存挙動があり、「デフォルト1で足すだけ」では済まないテーブルを個別に洗い出す。
6. **採番のテナント分離**: 請求書番号 `INV-…`/`WI-…` を会社ごとの番号空間に。
7. **フォールバックの時限性（審議で修正）**: companyId 未設定トークンの `companyId=1` フォールバックは「**テナントが1社しか存在しない間**」の時限措置。2社目テナント作成を**全セッション失効 + companyId クレーム必須化**のトリガーにする（残すと「クレーム欠落→自動的にテナント1のデータ」というセキュリティホールになる）。

### 2.3 取引関係（partner_links）

```
partner_links:
  id, requesterCompanyId, addresseeCompanyId,
  status: invited → accepted → suspended,
  requesterClientId,  -- A社のclientsのどの行がB社に対応するか
  addresseeClientId,  -- B社のclientsのどの行がA社に対応するか（B社が承認時に紐付け）
  invitedBy, acceptedBy, createdAt...
```

- A社が既存の取引先マスタ（`clients`）の1行を「システム連携に招待」→ B社管理者がリンク（トークン、genbaShares 方式）から承認 → 双方向リンク成立。
- **既存の clients テーブルは壊さない**。リンクは「clients 行 ↔ 相手テナント」の対応付けを外付けするだけ。未連携の取引先は今まで通り PDF 手渡し運用（後方互換）。
- **審議反映**: 同じ相手が「売上先」と「仕入先」で別マスタ行になる実務があるため、clientId 対応は partner_links 本体の1対1列ではなく **1:N 許容の対応テーブル（partner_link_client_maps）** に分離する。会社ペアの重複リンクは無順序ペア unique で防止。ライフサイクルに `rejected`（再招待可）を追加し、`suspended`（解除）後も**既存 submission は双方から閲覧可のまま**（証跡保全）と定義する。
- これで「A社がB社に請求書を提出している ＝ A社はB社のプロジェクトの協力会社」という関係が構造化され、B社側の管理画面の起点になる。

### 2.4 第1弾-a: 請求書提出 + 出面確認（会社間）

**作業員請求書の状態機械を会社間に写像する。**

```
partner_invoice_submissions:
  id, partnerLinkId,
  fromCompanyId, toCompanyId,
  invoiceId,            -- A社側の取引先請求書
  version, supersedesId, -- 再提出はイミュータブル新行（旧版は証跡として永続保全）
  billingPeriodFrom/To,  -- 請求対象期間（締め日ズレ対応: A社20日締め×B社末締めでも突合可能に）
  snapshotJson,         -- ★外部ビューDTO（ホワイトリスト方式）に通した凍結コピー
  status: submitted → received → under_review → approved / returned / superseded
  approvedAmount, adjustmentsJson, -- 査定・減額承認（申告額≠承認額を正規サポート）
  returnReason, reviewedBy, ...
  pdfKeys (請求書PDF / 出面確認表PDF / 添付証憑)  -- R2キーのみ
```

**審議で確定した設計要件（重大指摘の反映）**:
- **外部ビューDTO必須**: snapshotJson は請求書ビルダーの生 JSON をそのまま入れない。作業員名（請求ルール上「社内メモのみ・外部非表示」）・内部メモ・原価系フィールドを**構造的に含められないホワイトリストDTO**（`buildShareView` と同型）に通してから凍結する。genba.security.test 方式の「漏洩禁止フィールド総当たりテスト」を Phase 3 の完了条件に含める。
- **査定・減額承認**: 元請は協力会費・安全協力費・駐車場代等を控除して支払うのが常態（申告額=承認額はむしろ例外）。承認は `approvedAmount` + 控除明細（adjustments）付きの「金額修正付き承認」を第1弾から正規サポートし、買掛の自動起票は**承認額ベース**で行う。赤伝/相殺もこの adjustments で表現する。
- **再提出はイミュータブル**: 差戻し→修正は同一行の status 書き換えではなく、新スナップショット行 + `supersedesId` 参照。invoice ごとの active submission は unique 制約で常に1本。**B社が旧版を上位請求に取り込み済みの状態で A社が再提出したら、参照側に「参照元が更新されました」警告を伝播**する。多段チェーン（P3）の金額整合はこの仕組みで守る。

フロー（オーナーの実取引で言うと）:
1. **A社（充寵）**: 今まで通り月締め→取引先請求書V2生成。取引先が連携済みなら「システムで提出」ボタンが出る。提出時にスナップショット凍結（既存 `downloadPackage` の中身＝請求書+出面表+領収書+日報をパッケージとして相手受領箱へ）。
2. **B社（元請）**: 受領箱に着信。**出面確認画面**で A社申告の出面（スナップショット内の確認表）と、B社側が持つ情報（B社の genba/出面にA社作業員がいる場合はその記録）を**突合表示**。差分があれば行単位でハイライト。
3. B社が承認（必要なら査定・控除つき）→ A社側の invoice が承認済み扱いになり、B社側には**買掛（支払予定）レコードが承認額で自動起票**される。差戻し → 理由必須で A社に返る（作業員請求書の `returnInvoice` と同型）。※受領請求→支払予定→支払済みを管理する**受領側支払テーブル（partner_payables）は新規設計**（現行の入金は `invoices.receivedAmount` 相乗りで器がないため）。Phase 3 スコープに含める。
4. **多段チェーン**: B社は受領・承認した A社請求（外注費）を、B社自身の上位取引先への請求書に**原価参照として取り込める**（B社側の請求書ビルダーに「承認済み受領請求から取り込む」入口を追加）。四次→三次→二次→元請と同じ構造が再帰的に成立する。**税の再計算はしない**: 適格請求書の端数処理は請求書単位が原則のため、取り込みは金額をそのまま参照する「原価参照」であり、明細単位の税再計算による1円ズレを構造的に排除する。免税事業者混在時（仕入税額控除80%経過措置）の表示は将来課題として明記。
5. **入金/支払の対称化**: A社が「入金済み」にする操作と B社が「支払済み」にする操作を submission 経由で相互通知（片方の操作でもう片方に「相手が支払済みにしました」と表示。強制同期はしない＝柔軟性維持）。

**ミス削減ポイント**:
- 出面の突合が画面上で行単位に自動化 → 「電話で日数確認」「Excel突合」が消える。
- スナップショット凍結 → 「あとで数字が変わっていた」問題が構造的に起きない。
- 差戻し理由の記録 → 言った言わないが消える。
- 承認と同時に相手側へ買掛が起票 → 転記ミスゼロ。

### 2.5 第1弾-b: 作業員名簿・資格書提出（入場前グリーンファイル）

```
partner_roster_submissions:
  id, partnerLinkId, fromCompanyId, toCompanyId,
  projectRef (相手現場名 / 連携済みなら相手projectId),
  workerSetJson,        -- 提出時点の名簿スナップショット（employeeIdの配列+各人の名簿データ）
  pdfKeys,              -- generateRosterPdf 一式 + 資格証書ファイル（R2キー）
  status: submitted → received → registered / returned,
  perWorkerStatus,      -- 作業員単位の受理/差戻し（資格書不鮮明など）
  expiryAlerts          -- 資格・在留・保険の期限
```

1. **A社**: 既存の名簿PDF（`generateRosterPdf`）+ `qualifications`/`documents` の証書ファイルを選択し、対象現場を指定して提出。**既存資産がほぼそのまま使える**（新規開発は提出箱とUIのみ）。
2. **B社**: 受領箱で作業員ごとに確認 → 「登録済み」にすると、**B社側の genbaSiteWorkers に「他社所属の作業員」として自動登録**される（後述 2.6 のマッピング）。書類不備は作業員単位で差戻し。
3. **期限管理**: documents の `expiryDate`（在留カード・保険・資格）を提出先にも見せ、期限切れ接近を双方にアラート。**現場入場中に資格が失効する事故を構造的に防ぐ**。

### 2.6 ゲスト作業員 ⇔ 他社従業員のマッピング（平安電工パターンの核心）

現状: 元請の職人は A社 genba に `guestName`（文字列）でしか存在しない。

改修: `genbaSiteWorkers` に加算列
```
externalCompanyId   -- 所属テナント（partner_link 経由でのみ設定可）
externalEmployeeRef -- 相手テナント側の employeeId（直接参照ではなく提出スナップショット経由の写し）
ccusNumber          -- CCUS番号による会社横断の名寄せキー
```
- 名簿提出（2.5）を B社が受理すると、guestName 一致 or CCUS 一致で既存ゲスト行に**自動名寄せ**（曖昧なら管理者が手動マッチ）。文字列一致依存から ID 参照へ格上げ。
- これにより「A社の現場ビジョンにいる元請の職人」が「元請テナントの従業員」と繋がり、出面突合（2.4）の精度も上がる。
- 逆方向の genba 共有: 既存 `genbaShares`/`genbaWorkerLinks` を partner_link に紐付けるオプションを追加（「この共有リンクは元請テナントの管理画面に常設表示」）。**トークンの仕組み自体は変えない**ので既存リンクは無傷。

### 2.7 想定取引パターンの網羅（審議対象）

| # | パターン | 例 | 対応 |
|---|---|---|---|
| P1 | 標準下請け請求 | 三次→二次へ月次請求 | 2.4 の基本フロー |
| P2 | **管理代行+ゲスト混在**（オーナーの実取引） | 二次が元請現場を管理、元請職人がゲスト、二次→元請へ請求 | 2.4 + 2.6。出面は「A社管理のgenba出面」が突合の正になる点が P1 と逆 |
| P3 | 多段チェーン | 四次→三次→二次→元請 | 承認済み受領請求の再参照（2.4-4）。再帰構造なので追加開発なし |
| P4 | 片側非テナント | 相手が未導入 | 従来の PDF 手渡し + 将来: 閲覧専用リンク（genbaShares方式）で段階導入 |
| P5 | 双方向取引 | B社がA社に発注しつつ、別現場ではA社がB社に発注 | partner_link は方向を持たない対等リンク。submission が方向を持つ |
| P6 | 名簿だけ先行 | 請求連携はまだだが入場書類だけシステム化 | 2.5 は 2.4 と独立に動く設計 |
| P7 | 同一人物の二重所属 | 職人が複数社の応援に入る | CCUS 名寄せは「参照」のみで、雇用主テナントは常に1つ。二重請求検知は突合画面の警告に留める（柔軟性優先）。※警告文言は「他社経由でも請求されている」という他社情報の漏洩にならない表現に設計する |
| P8 | 支払通知書方式・出来高払い | 元請が査定して支払通知書を発行（下請は請求書を出さない）/ 出来高査定 | **第1弾は対象外**と明示。本プランの請求連携は「常用（日額×倍率）の月次請求」のみを対象とする。将来フェーズで検討 |

**締め日ズレ**: A社20日締め×B社末締めのような組合せが普通にあるため、submission は請求対象期間（billingPeriodFrom/To）を明示的に持ち、出面突合画面は**期間の部分重複を前提**に設計する（「月」単位の一致を仮定しない）。

## 3. リスク・エラー分析（審議結果）

### 3.1 データ漏洩リスク（最重要）
- **脅威**: フィルタ漏れ1箇所で他社の単価・原価・個人情報が見える。179 procedure 全部が対象。
- **対策**: (a) DB層で `companyId` を必須引数にする型設計（省略はコンパイルエラー）。(b) **クロステナント漏洩テスト**を Phase 1 の完了条件にする: 2社をシードし、A社セッションで全 list/get 系 API を叩いて B社データが0件であることを機械的に総当たり。(c) genba公開トークンは「トークンを知っている」ことが認可なので現状思想を維持しつつ、払い出しフィールドのホワイトリスト検査（既存 security.test の方式）を連携系にも適用。

### 3.2 請求金額の不一致
- **脅威**: A社の band 分け（深夜帯）とB社の検算がズレる。会社ごとに残業ルールが違う可能性。
- **対策**: 計算コア（`*V2Core.ts`）は純関数なので、**B社側の検算にも同じコアを使う**。ルール定数（`DAY_OT_REGULAR_CAP_TIMES10` 等）は将来テナント別設定に昇格（Phase 3）。それまでは「スナップショットの数字が正」で運用し、検算はあくまで参考表示。

### 3.3 出面突合の曖昧性
- **脅威**: guestName 表記ゆれ・同姓同名で突合ミス。
- **対策**: CCUS番号名寄せ + 手動マッチUI + 「未マッチ行は必ず警告表示」（黙って合致扱いにしない）。

### 3.4 既存本番への影響
- **脅威**: companyId バックフィルや unique 制約変更のマイグレーション事故。
- **対策**: 全マイグレーションは「列追加+デフォルト1+バックフィル」の加算のみで開始。unique 制約の付け替えは `MULTI_TENANT` 有効化の直前フェーズに隔離。migration-test CI（既存 `build.yml`）が守る。ロールバック手順を各フェーズで文書化。

### 3.5 スコープ暴走
- **脅威**: 通知基盤・課金・多言語など周辺要求で本丸が進まない。
- **対策**: 通知はアプリ内のみ（既存方針踏襲）。課金・オンボーディングは本プランの範囲外として明示的に切り離す。
