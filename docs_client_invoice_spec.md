# 取引先請求書（売上側）仕様メモ — 後フェーズ用（参考資料）

出典: 「取引先請求書の書き方・裁き方 参考資料」(PDF, 2026-06)。**確定仕様ではなく参考**。
実装は既存コード・実物請求書例・取引先ルール・税務確認に合わせて柔軟に。**今は未実装（後フェーズ）**。

## 役割（混同しない）
- 作業員請求書 = **支払い側**（会社→作業員）。取引先請求書 = **売上側**（会社→取引先）。
- 作業員請求書をそのまま取引先に出さない。会社側で**再集計**して作る。

## 構造（集計の基本軸）
**取引先ID × 対象月 × プロジェクトID × 請求単価グループ**
- 取引先 × 対象月 で1請求書。中をプロジェクト/現場ごとに分ける。
- **プロジェクトをまたいで合算しない**（同名「電気工事業A」でも別プロジェクトなら別行）。

## 電気工事業 A / B / C
- **作業員名ではなく「請求単価グループ」**。例: A=¥25,000 / B=¥21,000 / C=¥18,000。
- 同プロジェクト内で**同じ請求単価の作業員は合算**（作業員名は出さない）。**単価が違えば A/B/C に分ける**（1行に混ぜない）。
- 数量は「**日**」。「人工日」は使わない。

## 交通費・経費
- **取引先請求対象（isClientBillable）だけ**を**プロジェクト単位**で集計。作業員別・日別内訳は出さない。
- 除外: 自社負担 / **客先直接支払（二重請求しない）** / 集計対象外 / ゲスト。
- 必要な領収書をプロジェクト/請求書単位で添付。

## 残業・夜勤
- 取引先に請求する場合は日勤と**別行**（例: 夜勤 ¥32,000/日、残業 ¥3,500/時）。
- 請求有無・計算式は契約/実例に合わせ**固定しない**（設定・例外で吸収）。

## 消費税・源泉
- 小計・消費税・合計を表示。課税区分は実運用に合わせ固定しすぎない。
- **源泉徴収は支払い側（作業員請求書）の話。取引先請求書に混ぜない。**

## 標準で出さないもの
作業員名 / 作業員別・日別出勤 / 作業員別交通費内訳 / 作業員への支払単価 / 社内メモ。

## 集計ロジック（13手順の要約）
対象月→取引先→**締め完了済み**プロジェクト取得→参加者明細集計→同単価合算→単価グループでA/B/C行→別プロジェクトは別行→請求対象の交通費/経費をプロジェクト単位集計→自社負担/客先直払/対象外を除外→領収書紐付け→消費税→ドラフト生成。

## 実装方針（PDFの依頼＝スマートに）
- 機械的でなく、現行DB/API/画面に合わせた扱いやすい設計を提案 → 小さいPRで。
- **冪等**（同条件で何度作っても二重作成・二重計上しない）。集計キー= 取引先ID×対象月×プロジェクトID×請求単価グループ。
- 生成前に**プレビュー**（例外・不足・未確認を確認）。生成後は**ドラフトで編集可**。
- 税区分・残業・夜勤の未決は**設定/例外で吸収**（固定実装しない）。

## 既存コードとの接続（Phase E 着手時の土台）
- `server/invoiceBuilder.ts`：プロジェクト×クライアント単価から請求ドラフトを組む既存ロジック（売上側の出発点）。
- `employee_rates.clientRate`：請求単価（A/B/C の単価グループ源）。
- `getMonthlyClosingV2ClientTransportationBillingSummary`：**既に** isClientBillable=true かつ paid_by_client除外でプロジェクト単位集計済み（交通費の二重請求防止OK）。
- 月締めV2の「締め完了」現場を対象に集計。

## 絶対に混同しないこと
作業員請求書≠取引先請求書 / 支払単価≠請求単価 / プロジェクト跨ぎでA/B/C合算しない / 「日」表記（人工日禁止）/ 違う単価を1行に混ぜない / 支払元と取引先請求可否を混ぜない / 客先直接支払を二重請求しない / 作業員別・日別交通費を標準表示しない / 税・源泉・残業・夜勤を実例確認なしに固定しない。
