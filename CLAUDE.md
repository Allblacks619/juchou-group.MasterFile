# コーディング指針（一般・LLM 共通のミス削減）

一般的な LLM のコーディングミスを減らすための行動指針（出典: multica-ai/andrej-karpathy-skills）。上記のプロジェクト固有ルールと併用する。

**トレードオフ:** これらの指針は速度よりも慎重さを優先する。些細なタスクでは判断で使い分ける。

## 1. 実装前に考える（Think Before Coding）

**思い込まない。混乱を隠さない。トレードオフを明示する。**

実装前に:
- 前提を明示的に述べる。不確かなら聞く。
- 複数の解釈があるなら提示する。黙って1つを選ばない。
- よりシンプルな方法があるなら言う。必要なら押し返す。
- 不明点があれば止まる。何が不明かを名指しして、聞く。

## 2. シンプルさ優先（Simplicity First）

**問題を解く最小限のコード。投機的なものは書かない。**

- 依頼された以上の機能は作らない。
- 単一用途のコードに抽象化を持ち込まない。
- 依頼されていない「柔軟性」「設定可能性」を足さない。
- 起こり得ないシナリオへのエラー処理を書かない。
- 200行書いて50行にできるなら、書き直す。

自問: 「シニアエンジニアはこれを『複雑すぎる』と言うか？」 Yes ならシンプルにする。

## 3. 外科的な変更（Surgical Changes）

**触るべき所だけ触る。自分が散らかした分だけ片付ける。**

既存コードを編集するとき:
- 隣接するコード・コメント・整形を「改善」しない。
- 壊れていないものをリファクタしない。
- 自分ならこうする、と思っても既存のスタイルに合わせる。
- 無関係なデッドコードに気づいたら、消さずに指摘する。

自分の変更が孤児を生んだとき:
- 自分の変更で未使用になった import/変数/関数は削除する。
- 依頼がない限り既存のデッドコードは削除しない。

判定基準: 変更した全ての行が、ユーザーの依頼に直接たどれること。

## 4. ゴール駆動の実行（Goal-Driven Execution）

**成功基準を定義する。検証できるまでループする。**

タスクを検証可能なゴールに変換する:
- 「バリデーションを追加」→「不正入力のテストを書いてから、それを通す」
- 「バグを直す」→「再現するテストを書いてから、それを通す」
- 「X をリファクタ」→「変更前後でテストが通ることを保証する」

複数ステップのタスクでは、簡潔な計画を述べる:
```
1. [ステップ] → 検証: [チェック]
2. [ステップ] → 検証: [チェック]
3. [ステップ] → 検証: [チェック]
```

強い成功基準があれば自律的にループできる。弱い基準（「動くようにして」）は常に確認が必要になる。

---

**この指針が効いているサイン:** diff の不要な変更が減る、複雑化による書き直しが減る、ミスの後ではなく実装前に確認の質問が来る。
