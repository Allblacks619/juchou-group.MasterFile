# 作業員請求書 × 月締めV2 連携プラン（確定版）

最終更新: 2026-06-25

## 背景（業務フロー）

```
出面表(attendance) ─┐
                    ├─→ 作業員が月締め提出 ─→ ① 作業員請求書 を生成
単価管理(rate) ─────┘     (交通費・経費・出面)      ② 日報(作業報告) を生成
                                                         │
                                          請求書は「支払い待ち」
                                                         │
                                          こちらが支払う → 支払済として保存
```

- **月締め提出が請求書生成の前提**（提出が無いと生成不可）。
- **日報**＝請求書の中身の裏付け＋「いつ・どの現場に行ったか」を確認する書類。
- **出面表**と**単価管理**が請求額・日報・支払額の計算根拠。

## 確定した設計判断

1. **主軸**: 月締めV2（`monthly_closing_v2_*`）に繋ぎ替える。V1（`closing_submissions`）依存からの移行。
2. **労務費**: 出面日数 × 作業員支払単価を**自動計上、ただし作業員が編集可**。
3. **日報**: **作業員 × 月で1枚**（現場・日付ごとの内訳）。

## 重要な業務ルール（請求対象の判定）

- **労務費**: 出面表の「出勤扱い（`isWorkedType`）」レコードのみ。日数 = Σ round(hoursWorked / 8) を 10倍スケールで集計し、(daysTimes10/10) × 単価。
- **交通費・経費**: 月締めV2の経費明細（`monthly_closing_v2_expense_lines`）のうち、**作業員が立替えたもの（`paymentMethod === "paid_by_worker"`）のみ**を作業員請求書に計上。
  - 会社カード/ETC/取引先直払い等は作業員への支払対象外なので**除外**（記録としては残す）。
- **交通費の日割り計算（日報）**: 交通費は作業員が**現場ごとに提出した総額**を、その現場の**出面日数で日割り按分**し、その現場の出面日にだけ割り当てる。
  - 端数は最終出面日に乗せる（例：¥24,515 ÷ 14日 → ¥1,751×13 + ¥1,752）。実際の出面表の丸めと一致。
  - V2は交通費を現場×月の総額1件で保持（`allocationMethod="daily"`）。日報の日別表示はこの按分で再現する。
- **単価未設定**は請求書をエラーにせず、警告（warning）として可視化し、金額0で明細に出す（管理者が単価設定後に再生成）。

## 日報（作業報告）の理想フォーマット（実物の出面表に準拠）

- **作業員 × 月で1枚**。列は **日付 / 曜日 / 現場名 / 残業時間 / 交通費** のみ（余計な情報を省く）。
- 月のカレンダー全日を行に出し、出面のある日だけ現場名・残業・交通費を埋める。
- **夜勤は行で区別**（色分け／マーク）。集計は **昼勤出勤日数・夜勤出勤日数・残業時間合計**。
- 目的：締め提出と同時に生成され、管理者が**出勤状況をひと目で確認**できること。
- 出面の労務費・交通費が請求書と一致＝「いつ・どの現場・何日」の裏付け。

## 未確定（要・確認）

- **残業時間（残業○h）の請求単価ルール**：時間単価か、日勤単価からの割増か等。現状コアは未計上（日報には残業時間を表示）。
- **夜勤の単価**：`resolveWorkerPaymentRate` の night 単価で対応予定（夜勤レコードを別バケット集計）。
- **消費税・源泉徴収**：暫定で労務費10%/交通費・経費0%/源泉なし。実物の作業員請求書に合わせて要確認。

## 段階的実装プラン

- **Phase A — 元データをV2へ接続（土台）** ← 着手中
  - A-1: V2 から作業員請求書ドラフトを組み立てる純ロジック（`workerInvoiceV2Builder.ts`）＋単体テスト。
  - A-2: tRPC エンドポイント（`workerInvoice.getMyDraftV2` 等）でV2ドラフトを返す。既存V1は当面保持（フラグ切替）。
  - A-3: 永続化（`worker_invoices` / `worker_invoice_items`）への保存マッピング。
- **Phase B — 請求書の中身を完成**: 労務費＋交通費＋経費の自動計上、明細編集（単位/カテゴリ/税率）、PDFロゴ・社印。
- **Phase C — 日報（monthly_work_report）生成**: 出面表から作業員×月の内訳PDF。`monthly_closing_v2_generated_documents` に保存。
- **Phase D — 支払い管理連携**: 承認済み請求書→支払い待ち→支払済。`employee_payments` と連携。

## 非リグレッション制約（維持）

- 月締めを請求ルーターへ戻さない（エンドポイント移動なし）。
- 月締めフローから直接PDFを生成しない（ドラフト先行が必須挙動）。
- AppRates 修正・売上/支払分離モデルのロールバックなし。
- 既存のV1請求書フロー・closing reopen/編集復旧を壊さない（V2は並行追加）。
