# 作業員請求書 × 月締めV2 連携プラン（確定版）

最終更新: 2026-06-25

## 背景（業務フロー）

```
出面表(attendance) ─┐
                    ├─→ 作業員が月締め提出 ─→ ① 作業員請求書 を生成
単価管理(rate) ─────┘     (交通費・経費・出面)      ② 日報(作業報告) を生成
                                                         │
                                          請求書は「支払い待ち」
                                                         │
                                          こちらが支払う → 支払済として保存
```

- **月締め提出が請求書生成の前提**（提出が無いと生成不可）。
- **日報**＝請求書の中身の裏付け＋「いつ・どの現場に行ったか」を確認する書類。
- **出面表**と**単価管理**が請求額・日報・支払額の計算根拠。

## 確定した設計判断

1. **主軸**: 月締めV2（`monthly_closing_v2_*`）に繋ぎ替える。V1（`closing_submissions`）依存からの移行。
2. **労務費**: 出面日数 × 作業員支払単価を**自動計上、ただし作業員が編集可**。
3. **日報**: **作業員 × 月で1枚**（現場・日付ごとの内訳）。

## 現状のギャップ（着手前）

| 項目 | 現状 |
|---|---|
| 請求書の元データ | V1 `closing_submissions` 依存（`workerInvoice.getMyDraft`） |
| 請求書の中身 | 交通費＋経費のみ。労務費（出面×単価）が未計上 |
| 日報(monthly_work_report) | `monthly_closing_v2_generated_documents` の enum 枠のみ。生成ロジック未実装 |
| 支払い状態 | `employee_payments`(pending→confirmed→paid) はあるが請求書と未連携 |
| 出面×単価 | `employee_payments` 側では計算済み（`routers.ts:347`）。請求書側は未連携 |

## 重要な業務ルール（請求対象の判定）

- **労務費**: 出面表の「出勤扱い（`isWorkedType`）」レコードのみ。日数 = Σ round(hoursWorked / 8) を 10倍スケールで集計し、(daysTimes10/10) × 単価。
- **交通費・経費**: 月締めV2の経費明細（`monthly_closing_v2_expense_lines`）のうち、**作業員が立替えたもの（`paymentMethod === "paid_by_worker"`）のみ**を作業員請求書に計上。
  - 会社カード/ETC/取引先直払い等は作業員への支払対象外なので**除外**（記録としては残す）。
- **交通費の日割り計算（日報）**: 交通費は作業員が**現場ごとに提出した総額**を、その現場の**出面日数で日割り按分**し、その現場の出面日にだけ割り当てる。
  - 端数は最終出面日に乗せる（例：¥24,515 ÷ 14日 → ¥1,751×13 + ¥1,752）。実際の出面表の丸めと一致。
  - V2は交通費を現場×月の総額1件で保持（`allocationMethod="daily"`）。日報の日別表示はこの按分で再現する。
- **単価未設定**は請求書をエラーにせず、警告（warning）として可視化し、金額0で明細に出す（管理者が単価設定後に再生成）。

## 日報（作業報告）の理想フォーマット（実物の出面表に準拠）

- **作業員 × 月で1枚**。列は **日付 / 曜日 / 現場名 / 残業時間 / 交通費** のみ（余計な情報を省く）。
- 月のカレンダー全日を行に出し、出面のある日だけ現場名・残業・交通費を埋める。
- **夜勤は行で区別**（色分け／マーク）。集計は **昼勤出勤日数・夜勤出勤日数・残業時間合計**。
- 目的：締め提出と同時に生成され、管理者が**出勤状況をひと目で確認**できること。
- 出面の労務費・交通費が請求書と一致＝「いつ・どの現場・何日」の裏付け。

## 設定で吸収する（固定実装しない）＝引き継ぎメモ反映

実例・税理士確認の前に計算式を固定しない。明細・単価管理側で設定として持てる形にする。

- **残業（残業○h）**：請求に含めるか／日勤単価からの割増／残業専用時間単価——を**設定で選べる**ようにする。現状コアは未計上（日報には残業時間を表示）。デモの「未計上」は仮。
- **夜勤**：`resolveWorkerPaymentRate` の night 単価で**別行集計**（夜勤レコードを別バケット）。最終式は実例確認後。
- **消費税・源泉徴収**：明細ごとに「課税区分・税率・源泉対象・源泉率・表示方法」を持てる設計にする。デモの労務費10%/交通費・経費0%/源泉なしは**仮設定**。源泉は支払側（作業員請求書）のみ、取引先請求書に混ぜない。

## 引き継ぎメモ反映（業務ルールと境界）

- **役割を混同しない**：日報＝作業実績の裏付け／作業員請求書＝支払（会社→作業員）／取引先請求書＝売上（会社→取引先）。
- **月締めV2の主軸は 対象月 × 現場/プロジェクト**（worker-first へ戻さない）。作業員請求書・日報は支払側の per-worker 出力なので別物。月締め管理画面（/app/closings 等）は大きく触らない。
- **交通費**：内部は 作業員×月×現場 で保持。支払元（paymentMethod）と取引先請求可否（isClientBillable）は**別項目**。作業員請求書は `paid_by_worker` のみ（支払側）。取引先側は isClientBillable で別集計（売上側・別フェーズ）。取引先には作業員別・日別内訳を**標準表示しない**。
- **権限**：交通費の支払元／取引先請求可否／領収書／内部メモは admin・manager 以上のみ。**A-2 エンドポイントは API 側でアクセス制御**する。
- **用語**：「出勤日数」「〇日」を使う（出面件数・人日は使わない）。
- **ゲスト/集計対象外**：参加人数・請求集計・警告・バリデーションに含めない（管理者が理由を明記した場合のみ例外）。
- **売上側単価と支払側単価は別**：取引先請求は売上単価で現場単位に集計（同じ単価テーブルに無理にまとめない）。

## スコープ（メモ反映）

- **今やる**：Phase A（V2接続）→ B（請求書の中身・明細編集・PDF）→ C（日報PDF）。小さいPRで、保存・再読込・権限・集計を確認しながら。
- **後ろ倒し（今は本実装しない）**：支払い管理の本実装、最終締めロック、会計連携、外部決済、大規模UIリデザイン、/app/closings の大幅変更。
- **将来フェーズ**：取引先請求書（売上）の自動生成＝作業員請求書＋出面を**売上単価**で現場単位に集計。作業員別/日別内訳は標準非表示。

## 非リグレッション制約（維持）

- 月締めを請求ルーターへ戻さない（エンドポイント移動なし）。
- 月締めフローから直接PDFを生成しない（ドラフト先行が必須挙動）。
- AppRates 修正・売上/支払分離モデルのロールバックなし。
- 既存のV1請求書フロー・closing reopen/編集復旧を壊さない（V2は並行追加）。
